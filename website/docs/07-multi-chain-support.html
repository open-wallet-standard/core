07 — Multi-Chain Support

How LWS supports EVM, Solana, Cosmos, Bitcoin, Tron, and future chains through a plugin architecture.

Design Decision

LWS uses a plugin architecture where each blockchain family is supported by a self-contained chain plugin that implements a standard interface. The core runtime has zero chain-specific logic.

Why This Approach

ApproachProsCons
Built-in support for N chainsNo setup required; tightly integratedCore bloats with every chain; updates require new releases; cannot support long-tail chains
Plugin architectureCore stays minimal; chains added independently; community can contribute; versioned separatelySlightly more complex initial setup; plugin quality varies
Raw signing only (no chain awareness)Maximum simplicity; zero chain assumptionsCallers must build transactions themselves; no simulation; no address derivation; poor developer experience

Plugin Interface

Every chain plugin MUST implement the ChainPlugin interface. This is the contract between the LWS core runtime and any blockchain integration:

interface ChainPlugin {
  /** Unique chain family identifier, e.g. "evm", "solana", "cosmos" */
  readonly chainType: string;

  /** CAIP-2 chain IDs this plugin can handle */
  readonly supportedChains: string[];

  /** Elliptic curve used for key generation */
  readonly curve: "secp256k1" | "ed25519";

  /** SLIP-44 coin type for HD derivation */
  readonly coinType: number;

  /**
   * Derive a chain-native address from a public key.
   * @param publicKey - The raw public key bytes
   * @param chainId  - CAIP-2 chain identifier
   * @returns Chain-native address string
   */
  deriveAddress(publicKey: Uint8Array, chainId: string): string;

  /**
   * Build a chain-native transaction from a generic descriptor.
   * @param params - Transaction parameters (to, value, data, etc.)
   * @param chainId - CAIP-2 chain identifier
   * @returns Serialized unsigned transaction bytes
   */
  buildTransaction(
    params: Record<string, unknown>,
    chainId: string
  ): Promise<Uint8Array>;

  /**
   * Sign a serialized transaction with the given private key.
   * @param tx - Unsigned transaction bytes
   * @param privateKey - 32-byte private key
   * @param chainId - CAIP-2 chain identifier
   * @returns Signed transaction bytes ready for broadcast
   */
  sign(
    tx: Uint8Array,
    privateKey: Uint8Array,
    chainId: string
  ): Promise<Uint8Array>;

  /**
   * Sign an arbitrary message (EIP-191, Solana off-chain, etc.).
   * @param message - The message to sign
   * @param privateKey - 32-byte private key
   * @param chainId - CAIP-2 chain identifier
   * @returns Signature bytes
   */
  signMessage(
    message: Uint8Array,
    privateKey: Uint8Array,
    chainId: string
  ): Promise<Uint8Array>;

  /**
   * Broadcast a signed transaction to the network.
   * @param signedTx - Signed transaction bytes
   * @param chainId - CAIP-2 chain identifier
   * @returns Transaction hash
   */
  broadcast(
    signedTx: Uint8Array,
    chainId: string
  ): Promise<string>;

  /**
   * Simulate a transaction without broadcasting.
   * @param tx - Unsigned transaction bytes
   * @param chainId - CAIP-2 chain identifier
   * @returns Simulation result with gas estimate and state changes
   */
  simulate(
    tx: Uint8Array,
    chainId: string
  ): Promise<{
    success: boolean;
    gasEstimate: string;
    logs: string[];
    error?: string;
  }>;

  /**
   * Query the balance of an address.
   * @param address - Chain-native address
   * @param chainId - CAIP-2 chain identifier
   * @param asset - Optional asset identifier (defaults to native token)
   * @returns Balance result
   */
  getBalance(
    address: string,
    chainId: string,
    asset?: string
  ): Promise<BalanceResult>;
}

interface BalanceResult {
  /** Raw balance as a decimal string (smallest unit) */
  raw: string;
  /** Human-readable balance with decimals */
  formatted: string;
  /** Number of decimal places */
  decimals: number;
  /** Token symbol */
  symbol: string;
}

First-Party Plugins

LWS ships with five first-party plugins covering the most widely used blockchain ecosystems. Each plugin is published as a separate npm package and can be versioned independently.

EVM — @lws/plugin-evm

Supports Ethereum, Base, Arbitrum, Optimism, Polygon, and all EIP-155 compatible chains.

PropertyValue

SLIP-44 Coin Type60
Derivation Pathm/44'/60'/0'/0/{index}

Transaction Libraryethers v6 / viem (configurable)
Signing SchemeECDSA over secp256k1 + keccak256
RPC ProtocolJSON-RPC (Ethereum JSON-RPC spec)

Transaction Format

{
  "type": 2,                            // EIP-1559
  "chainId": 8453,                      // Base
  "nonce": 42,
  "to": "0x4B0897b0513fdC7C541B6d9D7E929C4e5364D2dB",
  "value": "1000000000000000000",       // 1 ETH in wei
  "data": "0x",
  "maxFeePerGas": "30000000000",        // 30 gwei
  "maxPriorityFeePerGas": "1000000000", // 1 gwei
  "gasLimit": "21000"
}

Simulation

The EVM plugin uses eth_estimateGas and eth_call for simulation. When the configured RPC supports it, the plugin also calls debug_traceCall to provide detailed state-change logs. Simulation is performed before every transaction unless explicitly disabled via simulate: false.

Commitment Mapping

Solana RPCs accept a commitment level. The plugin maps LWS confirmation levels to Solana commitments:

LWS ConfirmationSolana CommitmentUse Case
fastprocessedOptimistic reads; lowest latency
standardconfirmedDefault for most operations
finalizedfinalizedIrreversible; use for high-value transfers

Cosmos — @lws/plugin-cosmos

Supports Cosmos Hub, Osmosis, and any Cosmos SDK chain with standard Amino/Direct signing.

PropertyValue
Curvesecp256k1
SLIP-44 Coin Type118
Derivation Pathm/44'/118'/0'/0/{index}
Address FormatBech32 with chain-specific HRP (cosmos1..., osmo1...)
Transaction Library@cosmjs/stargate
Signing ModesSIGN_MODE_DIRECT (Protobuf) and SIGN_MODE_LEGACY_AMINO_JSON
RPC ProtocolTendermint RPC + gRPC-web

Address Derivation

Cosmos addresses are derived by taking the RIPEMD-160 hash of the SHA-256 hash of the compressed secp256k1 public key, then encoding with Bech32. The human-readable part (HRP) varies per chain:

cosmos1...   → Cosmos Hub
osmo1...     → Osmosis
juno1...     → Juno
stars1...    → Stargaze

The plugin maintains a registry mapping CAIP-2 chain IDs to their corresponding Bech32 HRP.

Bitcoin — @lws/plugin-bitcoin

Supports Bitcoin mainnet and testnet with native SegWit (bech32) addresses.

PropertyValue

Address FormatBech32 (bc1q... for mainnet, tb1q... for testnet)

Signing SchemeECDSA over secp256k1 + SHA-256d
UTXO ManagementBuilt-in coin selection (branch-and-bound, then largest-first fallback)

UTXO Handling

Unlike account-based chains, Bitcoin uses the UTXO model. The plugin automatically fetches unspent outputs for the address, selects inputs using an efficient coin-selection algorithm, and constructs the transaction with appropriate change outputs. The fee rate is estimated from the mempool or can be specified explicitly.

Tron — @lws/plugin-tron

Supports Tron mainnet and Shasta testnet.

Energy and Bandwidth

Tron uses an energy/bandwidth resource model instead of a simple gas fee. The plugin estimates energy and bandwidth consumption for each transaction and warns if the account lacks sufficient resources, in which case TRX will be burned to cover the cost.

Plugin Discovery

Plugins are discovered through two mechanisms:

Built-in Plugins

The five first-party plugins listed above are bundled with the default LWS installation and are loaded automatically. They require no additional configuration.

External Plugins

Community and third-party plugins are distributed as npm packages with the lws-plugin keyword in their package.json:

{
  "name": "@example/lws-plugin-sui",
  "version": "1.0.0",
  "keywords": ["lws-plugin"],
  "main": "dist/index.js",
  "lws": {
    "chainType": "sui",
    "supportedChains": [
      "sui:mainnet",
      "sui:testnet"
    ]
  }
}

After installation, external plugins are registered in ~/.lws/config.json:

{
  "plugins": {
    "builtin": ["evm", "solana", "cosmos", "bitcoin", "tron"],
    "external": [
      {
        "package": "@example/lws-plugin-sui",
        "version": "^1.0.0",
        "enabled": true
      },
      {
        "package": "@example/lws-plugin-aptos",
        "version": "^2.1.0",
        "enabled": true
      }
    ]
  }
}

External plugins are loaded in a sandboxed environment with limited filesystem access. They can only read their own configuration and communicate with the LWS runtime through the ChainPlugin interface. They MUST NOT have direct access to private keys — signing is delegated to the enclave, which calls the plugin's sign() method with the key material inside the isolated process.

RPC Configuration

Each chain requires an RPC endpoint for broadcasting transactions, simulating calls, and querying balances. RPC URLs are configured per chain in ~/.lws/config.json:

      "timeout": 30000,
      "retries": 3
    },
    "eip155:8453": {
      "url": "https://mainnet.base.org",
      "timeout": 30000,
      "retries": 3
    },
    "solana:5eykt4UsFv8P8NJdTREpY1vzqKqZKvdp": {
      "url": "https://api.mainnet-beta.solana.com",
      "timeout": 60000,
      "retries": 5
    },
    "cosmos:cosmoshub-4": {
      "url": "https://rpc.cosmos.network",
      "timeout": 30000,
      "retries": 3
    },
    "bip122:000000000019d6689c085ae165831e93": {
      "url": "https://blockstream.info/api",
      "timeout": 30000,
      "retries": 3
    },
    "tron:0x2b6653dc": {

}

If no RPC is configured for a chain, the plugin falls back to a hardcoded default public endpoint. Implementations SHOULD warn when using default endpoints in production, as they may be rate-limited or unreliable.

HD Derivation

LWS uses BIP-32 hierarchical deterministic key derivation from a single BIP-39 mnemonic seed. Each chain plugin defines its own derivation path according to BIP-44 and SLIP-44 standards. A single mnemonic can derive keys for every supported chain:

                          BIP-39 Mnemonic
                                |
                          Master Seed (512-bit)
                                |
              +-----------+-----+-----+-----------+-----------+
              |           |           |           |           |
         m/44'/60'   m/44'/501' m/44'/118'  m/84'/0'   m/44'/195'
          (EVM)      (Solana)   (Cosmos)   (Bitcoin)    (Tron)
              |           |           |           |           |
         0'/0/{i}    {i}'/0'    0'/0/{i}   0'/0/{i}   0'/0/{i}
              |           |           |           |           |
         0xab16...   7S3P4Hx...  cosmos1... bc1q7kw...  TJRab...

The derivation follows these standards:

  • BIP-32 — Defines the hierarchical deterministic wallet structure and key derivation algorithm.
  • BIP-39 — Defines the mnemonic phrase encoding (12 or 24 words) and seed generation.
  • BIP-44 — Defines the multi-account hierarchy: m / purpose' / coin_type' / account' / change / address_index.
  • BIP-84 — Defines the derivation path for native SegWit Bitcoin addresses (used instead of BIP-44 for Bitcoin).
  • SLIP-44 — The registered coin type table. Each chain family has a unique coin type number.

When a wallet is created from a mnemonic, the encrypted mnemonic entropy is stored once. Account addresses for any chain can be derived on demand without storing additional key material.

Adding a New Chain

Supporting a new blockchain requires zero changes to the LWS core. The process is:

  1. Implement the ChainPlugin interface — Create a new npm package that exports a class implementing all methods of the interface: deriveAddress, buildTransaction, sign, signMessage, broadcast, simulate, and getBalance.
  2. Register the CAIP-2 namespace — Define the supportedChains array with all CAIP-2 chain IDs the plugin handles. If the chain uses a new CAIP-2 namespace, document it in the plugin's README.
  3. Choose the derivation path — Look up the chain's registered SLIP-44 coin type. If none exists, request one from the SLIP-44 registry. Define the canonical derivation path.
  4. Publish and register — Publish the package to npm with the lws-plugin keyword. Users install it and add it to their config.json external plugins array.
  5. Test against the LWS plugin test suite — LWS provides a conformance test harness (@lws/plugin-test-utils) that validates every method of the ChainPlugin interface against known test vectors.
# Install the conformance test suite
npm install --save-dev @lws/plugin-test-utils

# Run conformance tests
npx lws-plugin-test --plugin ./dist/index.js
