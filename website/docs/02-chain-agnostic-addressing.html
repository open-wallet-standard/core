02 — Chain-Agnostic Addressing

How LWS identifies chains, accounts, and assets without being locked to any single ecosystem.

Design Decision

LWS uses CAIP (Chain Agnostic Improvement Proposal) identifiers as the canonical way to reference chains, accounts, and assets across every operation.

Why This Approach

ApproachProsCons
Chain-specific enums ("ethereum", "solana")Human-readableUnbounded set; no standard; ambiguous for testnets and L2s
Numeric chain IDs (1, 8453)Compact; widely used in EVMEVM-only; collisions across ecosystems
CAIP-2 / CAIP-10Universal; extensible; adopted by WalletConnect, Privy, x402More verbose; requires parsing

CAIP-2 Chain Identification

Every chain in LWS is identified by a CAIP-2 string with the format:

namespace:reference

The namespace identifies the blockchain family. The reference identifies the specific network within that family.

CAIP-10 Account Identification

Accounts are identified by combining the CAIP-2 chain ID with the account address:

chain_id:account_address

This creates a globally unique identifier for any account on any chain:

// EVM account on Base
eip155:8453:0xab16a96D359eC26a11e2C2b3d8f8B8942d5Bfcdb

// Solana account on mainnet
solana:5eykt4UsFv8P8NJdTREpY1vzqKqZKvdp:7S3P4HxJpyyigGzodYwHtCxZyUQe9JiBMHyRWXArAaKv

// Cosmos account on Osmosis
cosmos:osmosis-1:osmo1a2b3c4d5e6f7g8h9i0j...

CAIP-27 Method Invocation

When an agent invokes a signing operation, LWS uses the CAIP-27 pattern to route the request to the correct chain plugin:

{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "lws_signTransaction",
  "params": {
    "chainId": "eip155:8453",
    "request": {
      "method": "eth_signTransaction",
      "params": {
        "to": "0x4B0897b0513fdC7C541B6d9D7E929C4e5364D2dB",
        "value": "0xDE0B6B3A7640000",
        "gas": "0x5208"
      }
    }
  }
}

The chainId field determines which chain plugin handles the request. The inner request contains the chain-native method and parameters.

Asset Identification

Assets (tokens, NFTs) are identified by combining the CAIP-2 chain ID with the contract address:

chain_id:contract_address

Examples:

// USDC on Base
eip155:8453:0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913

// USDC on Ethereum
eip155:1:0xA0b86991c6218b36c1d19D4a2e9Eb0cE3606eB48

// Native ETH on Base (special "native" token)
eip155:8453:native

// SOL on Solana mainnet
solana:5eykt4UsFv8P8NJdTREpY1vzqKqZKvdp:native

The special token native refers to the chain's native gas token (ETH, SOL, ATOM, etc.). Implementations MUST recognize native as a reserved identifier that cannot be used as a contract address.

Interface Usage

CAIP identifiers appear throughout the LWS interface. Here are examples showing how they are used in core operations:

Signing a Transaction

const result = await lws.sign({
  walletId: "3198bc9c-6672-5ab3-d995-4942343ae5b6",
  chainId: "eip155:8453",           // CAIP-2: Base
  transaction: {
    to: "0x4B0897b0513fdC7C541B6d9D7E929C4e5364D2dB",
    value: "1000000000000000000",    // 1 ETH in wei
    data: "0x"
  }
});

Listing Accounts

const accounts = await lws.listAccounts({
  walletId: "3198bc9c-6672-5ab3-d995-4942343ae5b6"
});

// Returns:
// [
//   {
//     accountId: "eip155:8453:0xab16a96D...",    // CAIP-10
//     chainId: "eip155:8453",                     // CAIP-2
//     address: "0xab16a96D..."
//   }
// ]

Policy Rules

const policy: PolicyRule = {
  type: "chain_restriction",
  allowedChains: [
    "eip155:8453",                   // Base
    "eip155:1",                      // Ethereum mainnet
    "solana:5eykt4UsFv8P8NJdTREpY1vzqKqZKvdp"  // Solana mainnet
  ]
};

Shorthand Aliases

For CLI convenience, LWS supports optional shorthand aliases for well-known chains:

lws send --chain base --to 0xab16... --value 1.0
# Resolves to: eip155:8453

lws send --chain ethereum --to 0xab16... --value 1.0
# Resolves to: eip155:1

lws send --chain solana --to 7S3P4Hx... --value 1.0
# Resolves to: solana:5eykt4UsFv8P8NJdTREpY1vzqKqZKvdp

lws send --chain arbitrum --to 0xab16... --value 1.0
# Resolves to: eip155:42161

Aliases MUST be resolved to full CAIP-2 identifiers before any processing occurs. They are a presentation-layer convenience only. The SDK, API, and all internal operations use full CAIP identifiers exclusively. Implementations MUST reject unknown aliases with a clear error message.
