06 — Agent Access Layer

How AI agents, CLI tools, and applications access LWS wallets through MCP, REST, and library interfaces.

Design Decision

LWS exposes wallet functionality through three access modes: MCP (Model Context Protocol) for AI agents, a localhost REST API for programmatic access, and a native SDK for direct library integration. All three modes enforce the same key isolation and policy guarantees.

Why MCP First

The Model Context Protocol is emerging as the standard way for AI agents to interact with external tools. By implementing LWS as an MCP server, any MCP-compatible agent can use wallets without custom integration code.

MCP Server Tools

The LWS MCP server exposes the following tools. Each tool definition follows the MCP tool schema format with name, description, and inputSchema.

lws_list_wallets

List all wallets in the local vault, optionally filtered by chain type.

{
  "name": "lws_list_wallets",
  "description": "List all wallets in the local LWS vault. Returns wallet IDs, names, chain types, and account addresses. Never returns private keys.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "chainType": {
        "type": "string",
        "enum": ["evm", "solana", "bitcoin", "cosmos", "aptos", "sui"],
        "description": "Filter wallets by chain type. Omit to list all wallets."
      }
    },
    "required": []
  }
}

lws_get_balance

Get the balance of a wallet on a specific chain, including native and token balances.

{
  "name": "lws_get_balance",
  "description": "Get the balance of a wallet on a specific chain. Returns native token balance and optionally ERC-20/SPL token balances.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "walletId": {
        "type": "string",
        "description": "The wallet ID (UUID) to check balance for."
      },
      "chainId": {
        "type": "string",
        "description": "CAIP-2 chain identifier (e.g. 'eip155:1', 'solana:5eykt4UsFv8P8NJdTREpY1vzqKqZKvdp')."
      },
      "tokens": {
        "type": "array",
        "items": { "type": "string" },
        "description": "Optional list of token contract addresses to include in balance check."
      }
    },
    "required": ["walletId", "chainId"]
  }
}

lws_sign_transaction

Sign a transaction without broadcasting it. Returns the signed transaction bytes.

{
  "name": "lws_sign_transaction",
  "description": "Sign a transaction without broadcasting. Returns the signed transaction bytes. Policy engine is evaluated before signing. Simulation runs by default.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "walletId": {
        "type": "string",
        "description": "The wallet ID (UUID) to sign with."
      },
      "chainId": {
        "type": "string",
        "description": "CAIP-2 chain identifier."
      },
      "transaction": {
        "type": "object",
        "description": "Chain-specific transaction object (e.g. { to, value, data } for EVM)."
      },
      "simulate": {
        "type": "boolean",
        "description": "Whether to simulate before signing. Defaults to true.",
        "default": true
      }
    },
    "required": ["walletId", "chainId", "transaction"]
  }
}

lws_sign_and_send

Sign a transaction and broadcast it to the network. Returns the transaction hash.

{
  "name": "lws_sign_and_send",
  "description": "Sign a transaction and broadcast it to the network. Returns the transaction hash and optionally waits for confirmations. Policy engine is evaluated before signing.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "walletId": {
        "type": "string",
        "description": "The wallet ID (UUID) to sign with."
      },
      "chainId": {
        "type": "string",
        "description": "CAIP-2 chain identifier."
      },
      "transaction": {
        "type": "object",
        "description": "Chain-specific transaction object."
      },
      "confirmations": {
        "type": "number",
        "description": "Number of block confirmations to wait for. 0 returns immediately after broadcast.",
        "default": 1
      }
    },
    "required": ["walletId", "chainId", "transaction"]
  }
}

lws_sign_message

{
  "name": "lws_sign_message",
  "description": "Sign an arbitrary message or EIP-712 typed data structure. Used for authentication, off-chain signatures, and permit signatures.",
  "inputSchema": {
    "type": "object",
    "properties": {

        "type": "string",

MCP Configuration

To connect LWS as an MCP server, add it to your MCP client configuration.

Generic MCP Config

{
  "mcpServers": {
    "lws": {
      "command": "lws",
      "args": ["mcp", "serve"],
      "env": {}
    }
  }
}

Claude Code Config

Add to .claude/mcp.json in your project root or ~/.claude/mcp.json globally:

{
  "mcpServers": {
    "lws": {
      "command": "lws",
      "args": ["mcp", "serve"],
      "env": {},
      "alwaysAllow": [
        "lws_list_wallets",
        "lws_get_balance",
        "lws_simulate",
        "lws_get_policy"
      ]
    }
  }
}

The alwaysAllow array lists tools that do not require user confirmation. Read-only operations (list, balance, simulate, policy) are safe to auto-approve. Signing operations should require explicit user approval.

REST API

The LWS REST API provides HTTP access to the same wallet operations. It binds exclusively to 127.0.0.1 and is never exposed to the network.

Endpoints

MethodPathDescription
GET/v1/walletsList all wallets
POST/v1/walletsCreate a new wallet
GET/v1/wallets/:idGet wallet details
DELETE/v1/wallets/:idDelete a wallet
GET/v1/wallets/:id/balanceGet wallet balance
POST/v1/wallets/:id/signSign a transaction
POST/v1/wallets/:id/sign-and-sendSign and broadcast a transaction
POST/v1/wallets/:id/sign-messageSign an arbitrary message
POST/v1/wallets/:id/simulateSimulate a transaction
GET/v1/wallets/:id/policyGet wallet policies
PUT/v1/wallets/:id/policyUpdate wallet policies
DELETE/v1/wallets/:id/policy/:policyIdRemove a policy from a wallet

Authentication

The REST API uses Bearer token authentication. The token is stored in ~/.lws/config.json:

{
  "api": {
    "port": 2100,
    "bind": "127.0.0.1",
    "token": "lws_a1b2c3d4e5f6..."
  }
}

All requests must include the token in the Authorization header:

curl -X GET http://127.0.0.1:2100/v1/wallets \
  -H "Authorization: Bearer lws_a1b2c3d4e5f6..."

Because the API binds to 127.0.0.1 only, it is inaccessible from other machines. The token provides defense-in-depth against local privilege escalation and browser-based attacks (CSRF).

Library SDK

For direct integration, LWS provides a TypeScript/JavaScript SDK that communicates with the signing enclave in-process.

import { LWS } from "@lws/sdk";

// Initialize LWS (spawns enclave subprocess)
const lws = new LWS();

// Unlock a wallet
await lws.unlock("agent-treasury", { passphrase: "..." });

// List wallets
const wallets = await lws.listWallets({ chainType: "evm" });
console.log(wallets);
// [
//   {
//     id: "3198bc9c-...",
//     name: "agent-treasury",
//     chainType: "evm",
//     accounts: [
//       {
//         accountId: "eip155:8453:0xab16a96D...",
//         address: "0xab16a96D...",
//         chainId: "eip155:8453"
//       }
//     ]
//   }
// ]

// Sign and send a transaction
const result = await lws.signAndSend({
  walletId: "3198bc9c-...",
  chainId: "eip155:8453",
  transaction: {
    to: "0x4B0897b0513fdC7C541B6d9D7E929C4e5364D2dB",
    value: "1000000",
    data: "0x"
  },
  confirmations: 1
});
console.log(result.txHash);
// "0xabc123..."

// Lock wallet when done
await lws.lock("agent-treasury");

Agent Interaction Example

Here is a step-by-step walkthrough of an AI agent sending ETH via the MCP interface.

Step 1: List Wallets

The agent discovers available wallets by calling lws_list_wallets:

// Agent calls MCP tool
Tool: lws_list_wallets
Input: { "chainType": "evm" }

// LWS returns
{
  "wallets": [
    {
      "id": "3198bc9c-6672-5ab3-d995-4942343ae5b6",
      "name": "agent-treasury",
      "chainType": "evm",
      "accounts": [
        {
          "accountId": "eip155:8453:0xab16a96D359eC26a11e2C2b...",
          "address": "0xab16a96D359eC26a11e2C2b...",
          "chainId": "eip155:8453"
        }
      ]
    }
  ]
}

Step 2: Check Balance

The agent checks the wallet has sufficient funds:

// Agent calls MCP tool
Tool: lws_get_balance
Input: {
  "walletId": "3198bc9c-6672-5ab3-d995-4942343ae5b6",
  "chainId": "eip155:8453"
}

// LWS returns
{
  "native": {
    "symbol": "ETH",
    "balance": "1.5",
    "balanceWei": "1500000000000000000"
  }
}

Step 3: Simulate Transaction

The agent simulates the transaction to verify it will succeed:

// Agent calls MCP tool
Tool: lws_simulate
Input: {
  "walletId": "3198bc9c-6672-5ab3-d995-4942343ae5b6",
  "chainId": "eip155:8453",
  "transaction": {
    "to": "0x4B0897b0513fdC7C541B6d9D7E929C4e5364D2dB",
    "value": "1000000000000000",
    "data": "0x"
  }
}

// LWS returns
{
  "success": true,
  "gasEstimate": "21000",
  "gasPrice": "1000000000",
  "totalCost": "0.001021",
  "balanceAfter": "1.498979"
}

Step 4: Sign and Send

The agent signs and broadcasts the transaction. This step requires explicit user approval in MCP clients that enforce it:

// Agent calls MCP tool
Tool: lws_sign_and_send
Input: {
  "walletId": "3198bc9c-6672-5ab3-d995-4942343ae5b6",
  "chainId": "eip155:8453",
  "transaction": {
    "to": "0x4B0897b0513fdC7C541B6d9D7E929C4e5364D2dB",
    "value": "1000000000000000",
    "data": "0x"
  },
  "confirmations": 1
}

// LWS returns
{
  "txHash": "0x7f9e2c1d8a3b4e5f6a7b8c9d0e1f2a3b4c5d6e7f...",
  "blockNumber": 12345678,
  "confirmations": 1,
  "status": "confirmed"
}

Throughout this entire interaction, the agent never sees or handles any private key material. The signing enclave handles all cryptographic operations in an isolated subprocess.
