03 — Signing Interface

The core operations: signing, sending, simulating, and message signing.

Design Decision

LWS exposes a minimal, chain-agnostic interface with exactly four operations: sign(), signAndSend(), signMessage(), and simulate().

Rather than exposing dozens of chain-specific methods, LWS defines a small surface area that works identically across all supported chains. The chain-specific details are handled by plugins behind this interface.

Why This Approach

ProjectApproachTrade-off
PrivyCloud-based signing with embedded walletsRich features but requires network; not local-first
AgentKit (Coinbase)Action-based SDK with per-chain methodsEasy to use but tightly coupled to specific chains
Solana Wallet StandardFeature-based interface with signTransactionClean design but Solana-only
W3C Universal WalletDID-centric credential walletStandards-based but focused on credentials, not transactions

sign()

Signs a transaction and returns the signed payload without broadcasting it.

Request

interface SignRequest {
  walletId: WalletId;
  chainId: ChainId;                  // CAIP-2
  transaction: SerializedTransaction;
  simulate?: boolean;                // default: true
}

Response

interface SignResult {
  signedTransaction: string;         // hex or base64 encoded
  txHash: string;                    // transaction hash
  simulation?: SimulationResult;     // if simulate was true
}

Flow

  1. Resolve walletId to the encrypted wallet file.
  2. Validate chainId matches a chain the wallet supports.
  3. Evaluate all attached policies against the transaction (see 04 — Policy Engine).
  4. If simulate is true (default), run the transaction through the chain plugin's simulator.
  5. Decrypt the key material in an isolated process (see 05 — Key Isolation).
  6. Sign the transaction using the chain plugin's signer.
  7. Return the signed transaction and wipe the decrypted key from memory.

signAndSend()

Signs a transaction, broadcasts it to the network, and optionally waits for confirmation.

Request

interface SignAndSendRequest extends SignRequest {
  maxRetries?: number;               // broadcast retry count (default: 3)
  confirmations?: number;            // blocks to wait (default: 1)
}

Response

interface SignAndSendResult extends SignResult {
  status: "confirmed" | "pending" | "failed";
  blockNumber?: number;
  blockHash?: string;
  confirmations: number;
}

The flow is identical to sign() with two additional steps: after signing, the transaction is submitted to the chain plugin's RPC endpoint, then the implementation polls for confirmation up to the specified number of blocks.

signMessage()

Signs an arbitrary message for authentication or attestation purposes. This does NOT submit a transaction.

Request

interface SignMessageRequest {
  walletId: WalletId;
  chainId: ChainId;                  // CAIP-2
  message: string | Uint8Array;
  encoding?: "utf8" | "hex";         // default: "utf8"
}

Response

interface SignMessageResult {
  signature: string;                 // hex or base64 encoded
  messageHash: string;               // hash of the signed message
}

Chain-Specific Conventions

ChainStandardDescription
EVMEIP-191Personal sign with \x19Ethereum Signed Message:\n prefix
EVM (typed)EIP-712Structured data signing with domain separator
SolanaEd25519Direct Ed25519 signature over the message bytes
CosmosADR-036Arbitrary message signing using ADR-036 encoding

The chain plugin determines the correct signing convention based on the chainId namespace. EIP-712 typed data is passed as a JSON string in the message field.

simulate()

Simulates a transaction without signing or broadcasting. Used for pre-flight checks, gas estimation, and state change preview.

Request

interface SimulateRequest {
  walletId: WalletId;
  chainId: ChainId;                  // CAIP-2
  transaction: SerializedTransaction;
}

Response

interface SimulationResult {
  success: boolean;
  gasEstimate?: string;
  stateChanges: StateChange[];
  error?: string;
  warnings: string[];
}

interface StateChange {
  type: "balance_change" | "token_transfer" | "approval" | "contract_call";
  asset: string;                     // CAIP asset identifier or "native"
  from: string;
  to: string;
  amount?: string;
  description: string;               // human-readable summary
}

Simulation is powered by chain-specific plugins. For EVM chains, this typically uses eth_call and trace APIs. For Solana, it uses simulateTransaction. Implementations SHOULD return meaningful stateChanges to help agents and users understand the transaction's effects.

SerializedTransaction Format

The SerializedTransaction type is a union of chain-specific transaction formats. Each chain plugin defines its own shape:

type SerializedTransaction =
  | EvmTransaction
  | SolanaTransaction
  | CosmosTransaction;

interface EvmTransaction {
  to: string;
  value?: string;                    // wei amount
  data?: string;                     // hex-encoded calldata
  gas?: string;
  gasPrice?: string;
  maxFeePerGas?: string;
  maxPriorityFeePerGas?: string;
  nonce?: number;
}

interface SolanaTransaction {
  serializedMessage: string;         // base64-encoded transaction message
  feePayer?: string;
}

interface CosmosTransaction {
  messages: Array<{
    typeUrl: string;
    value: Record<string, unknown>;
  }>;
  fee?: {
    amount: Array<{ denom: string; amount: string }>;
    gasLimit: string;
  };
  memo?: string;
}

Error Handling

All operations return structured errors using a standard error interface:

interface LwsError {
  code: string;
  message: string;
  details?: Record<string, unknown>;
}

Concurrency

To prevent nonce conflicts and double-signing, LWS enforces a per-wallet mutex for all signing operations. When a sign() or signAndSend() call is in progress for a given wallet, subsequent calls for the same wallet are queued and executed sequentially.

// These calls for the SAME wallet execute sequentially:
const [tx1, tx2] = await Promise.all([
  lws.signAndSend({ walletId: "wallet-A", chainId: "eip155:8453", ... }),
  lws.signAndSend({ walletId: "wallet-A", chainId: "eip155:8453", ... }),
]);
// tx1 completes before tx2 begins — nonces are assigned in order.

// These calls for DIFFERENT wallets execute in parallel:
const [txA, txB] = await Promise.all([
  lws.signAndSend({ walletId: "wallet-A", chainId: "eip155:8453", ... }),
  lws.signAndSend({ walletId: "wallet-B", chainId: "eip155:8453", ... }),
]);

For EVM chains, the implementation MUST track the current nonce per account and increment it for each signed transaction. If a NONCE_CONFLICT error occurs, the implementation SHOULD refresh the nonce from the network and retry.
