08 — Wallet Lifecycle

How wallets are created, imported, exported, backed up, recovered, and migrated.

Design Decision

LWS provides complete lifecycle operations — creation, import, export, backup, recovery, and deletion — through both the CLI and the programmatic SDK, ensuring wallets can be managed by humans and agents alike.

Human Wallets vs. Agent Wallets

While the underlying storage format is identical, human and agent wallets have different lifecycle priorities:

OperationHuman WalletAgent Wallet
CreationInteractive; user writes down mnemonicProgrammatic; mnemonic stored encrypted, never displayed
BackupPaper seed phrase; hardware walletEncrypted vault backup; automated scheduled exports
RecoveryManual mnemonic entryAutomated restore from encrypted backup
MigrationRare; manual export/importCommon; move wallet between servers or environments
DeletionRarely neededFrequent; ephemeral agent wallets created and destroyed per task

Creation

Create from New Mnemonic

The most common creation path generates a new BIP-39 mnemonic, derives initial accounts, and stores everything encrypted in the vault.

CLI

lws wallet create \
  --name "agent-treasury" \
  --chain evm \
  --chains eip155:8453,eip155:1 \
  --accounts 3 \
  --password-env LWS_PASSWORD

TypeScript API

import { LWS } from "@lws/sdk";

const lws = new LWS();

const wallet = await lws.createWallet({
  name: "agent-treasury",
  chainType: "evm",
  chains: ["eip155:8453", "eip155:1"],
  accountCount: 3,
  password: process.env.LWS_PASSWORD,
});

// wallet.id          => "3198bc9c-6672-5ab3-d995-4942343ae5b6"
// wallet.accounts    => [{ accountId: "eip155:8453:0xab16...", ... }, ...]
// wallet.mnemonic    => undefined (never returned after creation)

Internal Flow

When createWallet is called, the following steps execute inside the key-isolation enclave:

  1. Generate entropy — 128 bits (12-word mnemonic) or 256 bits (24-word mnemonic) from a cryptographically secure random source (crypto.getRandomValues).
  2. Encode mnemonic — Convert entropy to a BIP-39 mnemonic phrase with checksum.
  3. Derive master seed — Apply PBKDF2-HMAC-SHA512 with the mnemonic and optional passphrase to produce a 512-bit master seed.
  4. Derive account keys — For each requested account index, derive the private key using the chain plugin's derivation path (e.g., m/44'/60'/0'/0/0).
  5. Derive addresses — Call the chain plugin's deriveAddress() with each public key.
  6. Encrypt mnemonic — Encrypt the mnemonic entropy with AES-256-GCM using a key derived from the password via scrypt.
  7. Write wallet file — Persist the encrypted wallet JSON to ~/.lws/wallets/<wallet-id>.json with 0600 permissions.
  8. Clear sensitive memory — Zero out mnemonic, seed, and private key buffers. Return only the public wallet descriptor.

Create from Existing Private Key

For importing a single private key (e.g., from an existing setup):

CLI

lws wallet create \
  --name "imported-key" \
  --chain evm \
  --private-key-env PRIVATE_KEY \
  --password-env LWS_PASSWORD

TypeScript API

const wallet = await lws.createWallet({
  name: "imported-key",
  chainType: "evm",
  privateKey: process.env.PRIVATE_KEY,
  password: process.env.LWS_PASSWORD,
});

// wallet.keyType => "private_key" (not "mnemonic")

When created from a private key, the wallet's key_type is set to "private_key". This wallet can only derive a single account and cannot be extended to other chains or additional accounts.

Import

LWS supports importing wallets from multiple external formats.

Keystore v3

Import a standard Ethereum Keystore v3 JSON file:

lws wallet import \
  --format keystore-v3 \
  --file /path/to/keystore.json \
  --source-password "original-password" \
  --password-env LWS_PASSWORD

The wallet is decrypted with the source password, then re-encrypted with the LWS vault password using AES-256-GCM. The original keystore file is not modified.

BIP-39 Mnemonic

Import an existing mnemonic phrase:

lws wallet import \
  --format mnemonic \
  --chain evm \
  --chains eip155:8453,eip155:1 \
  --accounts 5 \
  --password-env LWS_PASSWORD

The CLI will prompt interactively for the mnemonic phrase via a secure input (no echo). The phrase is validated against the BIP-39 wordlist and checksum before proceeding.

WIF (Bitcoin)

Import a Bitcoin private key in Wallet Import Format:

lws wallet import \
  --format wif \
  --chain bitcoin \
  --wif-env BTC_WIF \
  --password-env LWS_PASSWORD

Solana Keypair JSON

Import a Solana CLI keypair file (the 64-byte JSON array format used by solana-keygen):

lws wallet import \
  --format solana-keypair \
  --file ~/.config/solana/id.json \
  --password-env LWS_PASSWORD

The 64-byte array is split into the 32-byte private key and 32-byte public key. The public key is verified to match the derived address before storing.

Export

LWS supports exporting wallet key material in several formats. All export operations require password authentication and produce audit log entries.

Mnemonic Export

Warning: Exporting a mnemonic exposes the master key material for all accounts across all chains derived from this wallet. Handle with extreme care.

CLI

lws wallet export \
  --wallet-id 3198bc9c-... \
  --format mnemonic \
  --password-env LWS_PASSWORD

# Output (to stdout):
# abandon abandon abandon abandon abandon abandon
# abandon abandon abandon abandon abandon about

TypeScript API

const mnemonic = await lws.exportWallet({
  walletId: "3198bc9c-...",
  format: "mnemonic",
  password: process.env.LWS_PASSWORD,
});

// mnemonic => "abandon abandon abandon ..."
// SECURITY: Clear this value from memory immediately after use

Implementations MUST log a warning to both the console and the audit log when a mnemonic is exported. The audit log entry MUST NOT contain the mnemonic itself.

Keystore v3 Export

Export a wallet as a standard Ethereum Keystore v3 file, compatible with MetaMask, Geth, and other tools:

lws wallet export \
  --wallet-id 3198bc9c-... \
  --format keystore-v3 \
  --output /path/to/exported-keystore.json \
  --password-env LWS_PASSWORD \
  --export-password-env EXPORT_PASSWORD

The exported file uses the EXPORT_PASSWORD for encryption, which can differ from the vault password. Only single-key wallets (key_type: "private_key") or the first derived key from a mnemonic wallet can be exported in this format.

Raw Private Key Export

Warning: Raw private keys are the most dangerous export format. They are unencrypted and grant full control over the account.

lws wallet export \
  --wallet-id 3198bc9c-... \
  --format raw \
  --account-index 0 \
  --password-env LWS_PASSWORD

# Output: 0x4c0883a69102937d623...

Raw key export MUST require an explicit --confirm-unsafe flag or interactive confirmation. Implementations SHOULD refuse raw export over network interfaces (REST API, MCP) and restrict it to local CLI usage only.

Backup

LWS supports full vault backup for disaster recovery. Backups include all wallet files, policy definitions, and configuration — but NOT the audit log (which may be very large).

Full Vault Backup

lws vault backup \
  --output /path/to/backup.lws.tar.gz.enc \
  --password-env LWS_PASSWORD

The backup process:

  1. Creates a tar.gz archive of ~/.lws/wallets/, ~/.lws/policies/, and ~/.lws/config.json.
  2. Encrypts the archive with AES-256-GCM using a key derived from the vault password.
  3. Writes the encrypted archive to the specified output path.
  4. Verifies the backup by performing a trial decryption.

Restore from Backup

lws vault restore \
  --input /path/to/backup.lws.tar.gz.enc \
  --password-env LWS_PASSWORD \
  --target ~/.lws

Restore decrypts the archive and writes the wallet files, policies, and configuration to the target directory. If the target directory already contains wallets, the CLI will prompt for confirmation before overwriting. Existing wallets not present in the backup are preserved.

Automated Backup

For agent environments, automated periodic backups can be configured in ~/.lws/config.json:

{
  "backup": {
    "enabled": true,
    "schedule": "0 2 * * *",
    "destination": "/mnt/secure-backup/lws/",
    "retention": {
      "daily": 7,
      "weekly": 4,
      "monthly": 6
    },
    "encryption": {
      "method": "aes-256-gcm",
      "keySource": "vault-password"
    },
    "notify": {
      "onSuccess": false,
      "onFailure": true,
      "webhook": "https://hooks.example.com/backup-alerts"
    }
  }
}

Recovery

From Mnemonic

A wallet can be fully recovered from its BIP-39 mnemonic. The recovery process re-derives all accounts and scans the blockchain for activity:

lws wallet recover \
  --chain evm \
  --chains eip155:8453,eip155:1 \
  --gap-limit 20 \
  --password-env LWS_PASSWORD

The CLI prompts for the mnemonic phrase via secure input. Recovery uses a gap limit (default: 20) — it derives consecutive account addresses and scans for on-chain activity. If 20 consecutive addresses show no transactions, scanning stops. All accounts with activity are added to the recovered wallet.

From Backup

Vault-level recovery uses the lws vault restore command described in the Backup section above. This is the preferred recovery method for agent wallets, as it restores not just keys but also policies, configuration, and plugin settings.

Deletion

Wallets can be permanently deleted from the vault:

lws wallet delete \
  --wallet-id 3198bc9c-... \
  --password-env LWS_PASSWORD \
  --confirm

The deletion process:

  1. Authenticate — Verify the password against the wallet's encryption.
  2. Check balances — Query all accounts for remaining balances. If non-zero balances are found, display a warning and require --force or interactive confirmation.
  3. Remove policy bindings — Detach any policies that reference this wallet.
  4. Secure delete — Overwrite the wallet file with random bytes, then delete it. On supported filesystems, use shred or equivalent.
  5. Audit log — Record the deletion event (wallet ID, timestamp, account addresses) in the audit log. The audit entry persists even after the wallet file is gone.

Deletion is irreversible. If the mnemonic or a backup is not available, all keys in the wallet are permanently lost. Implementations MUST make this clear to the user before proceeding.

Key Rotation

LWS provides a convenience command to rotate keys by creating a new wallet and optionally transferring assets from the old wallet:

lws wallet rotate \
  --wallet-id 3198bc9c-... \
  --password-env LWS_PASSWORD \
  --transfer-assets \
  --delete-old

The rotation process:

  1. Create a new wallet with the same chain type and account structure as the old one.
  2. Query balances on all accounts of the old wallet.
  3. Build transfer transactions from each old account to the corresponding new account (same index). For ERC-20 tokens, the plugin enumerates known token balances.
  4. Execute transfers with policy checks and simulation.
  5. Delete the old wallet (if --delete-old is specified) after all transfers are confirmed.

Key rotation is not atomic across chains. If a transfer fails on one chain, the process halts and reports which transfers succeeded and which need manual intervention. The old wallet is NOT deleted if any transfer fails.

Wallet Discovery

Agents and CLI users can discover wallets in the vault by querying with various criteria:

TypeScript API

// List all wallets
const all = await lws.listWallets();

// Find wallets by chain type
const evmWallets = await lws.listWallets({
  chainType: "evm",
});

// Find wallets with a specific account on Base
const baseWallets = await lws.listWallets({
  chainId: "eip155:8453",
});

// Find wallet by name
const treasury = await lws.listWallets({
  name: "agent-treasury",
});

// Get full wallet descriptor (without key material)
const wallet = await lws.getWallet("3198bc9c-...");

// wallet => {
//   id: "3198bc9c-...",
//   name: "agent-treasury",
//   chainType: "evm",
//   createdAt: "2026-02-27T10:30:00Z",
//   accounts: [...],
//   policyIds: ["spending-limit-01"],
//   keyType: "mnemonic",
//   metadata: {}
// }

CLI

# List all wallets
lws wallet list

# Filter by chain
lws wallet list --chain evm

# Show detailed info for a specific wallet
lws wallet info --wallet-id 3198bc9c-...

# Output:
# Wallet: agent-treasury (3198bc9c-...)
# Chain:  evm
# Type:   mnemonic
# Created: 2026-02-27T10:30:00Z
# Accounts:
#   [0] eip155:8453:0xab16a96D...  (Base)
#   [1] eip155:1:0xab16a96D...     (Ethereum)
#   [2] eip155:42161:0xab16a96D... (Arbitrum)
# Policies: spending-limit-01

Lifecycle State Diagram

Every wallet in LWS exists in one of four states:

                    +------------------+
                    |                  |
        +-----------+   Create (new)   |
        |           |                  |
        |           +------------------+
        |
        |           +------------------+
        |           |                  |
        +-----------+ Import (existing)|
        |           |                  |
        |           +------------------+
        |
        |           +------------------+
        |           |                  |
        +-----------+ Recover (mnemonic|
        |           |   or backup)     |
        |           +------------------+
        |
        v
  +------------+        lock         +------------+
  |            | ------------------> |            |
  |   Active   |                     |   Locked   |
  |            | <------------------ |            |
  +------------+       unlock        +------------+
        |                                   |
        |   +------------+                  |
        +-->|            |                  |
        |   |   Export   |                  |
        |   |   Backup   |                  |
        |   +------------+                  |
        |                                   |
        v                                   v
  +------------+                     +------------+
  |            |                     |            |
  |  Deleted   |                     |  Deleted   |
  |            |                     |            |
  +------------+                     +------------+
  • Active — The wallet is decrypted in the enclave and available for signing operations. This is the normal operating state.
  • Locked — The wallet file exists but no decrypted key material is held in memory. The wallet must be unlocked (password provided) before any signing operation. Wallets enter this state on startup or after an inactivity timeout.
  • Export / Backup — A side operation that can be performed from the Active state. The wallet remains active after export or backup.
  • Deleted — The wallet file has been securely erased. This state is terminal and irreversible.
