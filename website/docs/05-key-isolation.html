05 — Key Isolation

How LWS prevents private keys from leaking to agents, LLMs, logs, or parent processes.

Design Decision

LWS uses subprocess isolation: a dedicated signing enclave process holds decrypted key material, and the parent process (agent, CLI, app) communicates with it exclusively over a Unix domain socket. The parent never sees the private key.

Isolation Approach Comparison

ApproachSecurityPerformanceComplexityUsed By
In-process (key in memory) Low — any library or LLM context can read the key Fastest Trivial ethers.js Wallet, viem Account
TEE enclaves (SGX, TrustZone) Very high — hardware-attested isolation Near-native Very high — requires specific hardware, attestation infra Privy, Phala, Turnkey
MPC / threshold signing High — no single party holds full key Slow (multi-round) Very high — distributed protocol, key resharing Lit Protocol, Fireblocks
Subprocess isolation High — OS process boundary, no shared memory Fast (~1ms IPC overhead) Low — standard OS primitives LWS, ssh-agent, gpg-agent

Subprocess isolation provides strong security guarantees using only standard OS primitives. It requires no special hardware, no network connectivity, and no multi-party coordination. The same model has been proven in production by ssh-agent and gpg-agent for decades.

The parent process spawns the signing enclave as a child process at startup. The enclave:

  1. Creates a Unix domain socket at a randomized path under /tmp/lws-<random>/enclave.sock
  2. Sets socket permissions to 0600 (owner-only)
  3. Drops all capabilities except signing
  4. Optionally applies seccomp or pledge filters to block network, filesystem, and exec syscalls

Enclave Protocol

Communication between the parent and enclave uses JSON-RPC 2.0 over a Unix domain socket. The protocol is deliberately minimal.

Request Format

{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "sign",
  "params": {
    "wallet_id": "3198bc9c-6672-5ab3-d995-4942343ae5b6",
    "chain_id": "eip155:8453",
    "payload": "0x02f87083014a34...",
    "encoding": "hex"
  }
}

Response Format

{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "signature": "0x3045022100...",
    "public_key": "0x04a1b2c3d4...",
    "encoding": "hex"
  }
}

Error Response

{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32001,
    "message": "wallet is locked",
    "data": { "wallet_id": "3198bc9c-..." }
  }
}

Enclave Methods

MethodDescriptionParameters
sign Sign a transaction payload wallet_id, chain_id, payload, encoding
sign_message Sign an arbitrary message wallet_id, chain_id, message, encoding
unlock Decrypt a wallet’s key material into memory wallet_id, passphrase
lock Zero and release a wallet’s decrypted key material wallet_id
status Return the lock state of all wallets (none)

Key Lifecycle Within the Enclave

The enclave follows a strict sequence for handling key material:

  1. Receive request — The enclave receives a JSON-RPC sign or sign_message request over the Unix socket.
  2. Verify wallet is unlocked — Check that the requested wallet_id has been previously unlocked. If not, return error code -32001.
  3. Retrieve key from mlock’d region — The decrypted key material is held in memory pages locked with mlock(2) to prevent swapping to disk.
  4. Derive signing key — If the wallet stores a mnemonic, derive the chain-specific private key using the appropriate BIP-44 derivation path.
  5. Hash the payload — Apply the chain-appropriate hashing algorithm (e.g., Keccak-256 for EVM, SHA-256 for Bitcoin).
  6. Produce signature — Sign the hash using the chain-appropriate algorithm (e.g., ECDSA secp256k1 for EVM, Ed25519 for Solana).
  7. Zero derived key immediately — If a derived key was created in step 4, it is overwritten with zeros and freed immediately after signing. Only the master key/mnemonic persists in the mlock’d region.
  8. Return signature — The signature and public key are returned to the parent. No key material ever crosses the socket boundary.

Passphrase Handling

The unlock method requires a passphrase to decrypt the wallet. LWS supports three delivery methods:

1. Interactive Prompt

The default method. The enclave opens /dev/tty directly and reads the passphrase, bypassing stdin entirely. This ensures the passphrase never flows through the parent process.

$ lws unlock agent-treasury
Enter passphrase for agent-treasury: ********
Wallet unlocked.

2. Environment Variable

For automated environments, the passphrase can be passed via an environment variable set only on the enclave process:

// Parent spawns enclave with passphrase in env
spawn("lws-enclave", {
  env: { LWS_PASSPHRASE: passphrase }
})
// Parent immediately clears its own copy
passphrase = undefined;

The enclave reads LWS_PASSPHRASE, copies it to mlock’d memory, and immediately overwrites the environment variable.

3. File Descriptor

The most secure automated method. The parent writes the passphrase to a pipe, and the enclave reads it from the inherited file descriptor:

// Parent creates pipe and passes read-end FD to enclave
const [readFd, writeFd] = pipe();
spawn("lws-enclave", { stdio: ["pipe", "pipe", "pipe", readFd] });
write(writeFd, passphrase);
close(writeFd);
// Passphrase only exists momentarily in kernel pipe buffer

Threat Model

ThreatMitigation
Agent exfiltration — LLM or agent code attempts to read private keys Keys exist only in the enclave process. The parent (where agents run) has no access to key memory.
Parent memory read — attacker reads parent process memory No key material exists in parent memory. Only wallet IDs, public addresses, and signatures.
Logging / LLM context leak — keys accidentally logged or sent to LLM Keys never enter the parent process, so they cannot be logged or included in LLM prompts.
Core dump — crash produces memory dump containing keys Enclave disables core dumps via prctl(PR_SET_DUMPABLE, 0) on Linux. On macOS, PT_DENY_ATTACH is used.
Swap to disk — OS swaps key-containing pages to disk All key material is held in mlock(2)’d pages that the OS cannot swap.
Cold boot attack — attacker reads RAM after power loss Partially mitigated by zeroing keys on lock. Full mitigation requires hardware (TEE) or encrypted RAM.
Compromised enclave binary — attacker replaces the signing binary Binary integrity checks on startup. Implementations SHOULD verify the enclave binary hash against a known value.
Brute-force passphrase — attacker tries passphrases offline scrypt with N=262144, r=8, p=1 makes brute-force extremely expensive (~1 second per attempt on modern hardware).

Defense in Depth

Subprocess isolation is the baseline. LWS defines extension points for additional security layers, all communicating through the same enclave protocol:

1. OS-Level Sandboxing

On supported platforms, the enclave restricts its own syscalls:

  • Linux: seccomp-bpf — allow only read, write, mmap, mlock, munlock, close, exit_group, and crypto-related syscalls.
  • OpenBSD: pledge("stdio") — restrict to standard I/O only.
  • macOS: sandbox-exec profile denying network and filesystem access.

2. TEE Backends

On systems with Trusted Execution Environments, the enclave can run inside a TEE:

  • Intel SGX: Enclave runs in an SGX enclave with remote attestation.
  • ARM TrustZone: Key material stored in the secure world, inaccessible from the normal world.
  • Apple Secure Enclave: Key generation and signing delegated to the Secure Enclave Processor (SEP).

3. Hardware Wallet Delegation

The enclave can delegate signing to an external hardware wallet:

  • Ledger / Trezor: The enclave forwards the sign request to the connected hardware wallet via USB/HID. The hardware wallet holds the key and produces the signature.
  • The enclave protocol remains identical — the parent process does not know or care whether signing happens in software or hardware.

4. Key Sharding (Shamir’s Secret Sharing)

For high-value wallets, the encrypted key can be split into n shares with a threshold of k:

  • Shares can be distributed across multiple machines, USB drives, or trusted parties.
  • The enclave reconstructs the key from k shares at unlock time, then immediately discards the individual shares.
  • This adds offline redundancy without requiring MPC infrastructure.

LWS is the only approach that provides strong key isolation without requiring network connectivity, cloud infrastructure, or specialized hardware. When hardware is available (TEEs, hardware wallets), LWS can leverage it as an additional layer.
