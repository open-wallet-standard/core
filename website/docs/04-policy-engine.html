04 — Policy Engine

How transaction policies are defined, evaluated, and enforced before any key material is touched.

Design Decision

LWS enforces all policy checks before decrypting key material. The policy engine acts as a pre-signing gate — if a transaction violates any rule, the signing process never begins and the private key is never loaded into memory.

Why This Approach

ApproachProsCons
Application-layer checks (SDK validation)Simple to implementBypassable; no enforcement guarantee; keys already decrypted
Smart contract guards (on-chain)Tamper-proof; verifiableGas costs; chain-specific; cannot prevent signing itself
Pre-signing gate (policy engine)Keys never touched if denied; chain-agnostic; auditableMust trust the local runtime

Policy Structure

A policy is a JSON document containing an array of rules. Each rule is evaluated independently, and all rules must pass for the transaction to proceed.

{
  "id": "spending-limit-01",
  "name": "Agent Treasury Policy",
  "rules": [
    {
      "type": "max_value",
      "chainId": "eip155:8453",
      "asset": "native",
      "maxAmount": "1000000000000000000",
      "period": "24h"
    },
    {
      "type": "contract_allowlist",
      "contracts": [
        "eip155:8453:0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913",
        "eip155:8453:0x4200000000000000000000000000000000000006"
      ]
    },
    {
      "type": "require_simulation"
    },
    {
      "type": "chain_restriction",
      "allowedChains": [
        "eip155:8453",
        "eip155:1"
      ]
    }
  ],
  "action": "deny"
}

Rule Types

max_value — Spending Limits

Limits the total value that can be sent within a rolling time period. The maxAmount is denominated in the asset's smallest unit (wei for ETH, lamports for SOL).

{
  "type": "max_value",
  "chainId": "eip155:8453",
  "asset": "native",
  "maxAmount": "1000000000000000000",
  "period": "24h"
}

Supported period values: "1h", "24h", "7d", "30d". If period is omitted, the limit applies as a lifetime cap. The implementation tracks cumulative spending in a local state file (see Spending State below).

allowlist — Allowed Addresses

Only permits transactions to addresses in the list. If the to address is not in the allowlist, the transaction is denied.

{
  "type": "allowlist",
  "addresses": [
    "0x4B0897b0513fdC7C541B6d9D7E929C4e5364D2dB",
    "0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913"
  ]
}

denylist — Blocked Addresses

Blocks transactions to any address in the list. If the to address is in the denylist, the transaction is denied immediately.

{
  "type": "denylist",
  "addresses": [
    "0xKnownScamAddress...",
    "0xSanctionedAddress..."
  ]
}

chain_restriction — Allowed Chains

Restricts the wallet to only operate on specified chains. Any transaction targeting a chain not in the list is denied.

{
  "type": "chain_restriction",
  "allowedChains": [
    "eip155:8453",
    "eip155:1",
    "solana:5eykt4UsFv8P8NJdTREpY1vzqKqZKvdp"
  ]
}

contract_allowlist — Allowed Contracts

Limits contract interactions to a known set of verified contracts. Uses CAIP asset identifiers to scope contracts to specific chains.

{
  "type": "contract_allowlist",
  "contracts": [
    "eip155:8453:0x833589fCD6eDb6E08f4c7C32D4f71b54bdA02913",
    "eip155:8453:0x4200000000000000000000000000000000000006",
    "eip155:1:0xA0b86991c6218b36c1d19D4a2e9Eb0cE3606eB48"
  ]
}

require_simulation — Mandatory Simulation

Forces every transaction to be simulated before signing. If simulation fails or returns warnings, the transaction is denied.

{
  "type": "require_simulation"
}

time_restriction — Time-Based Access

Restricts signing operations to specific hours of the day. Useful for limiting agent activity to business hours.

{
  "type": "time_restriction",
  "allowedHours": {
    "start": 9,
    "end": 17,
    "timezone": "America/New_York"
  }
}

custom — Custom Evaluator

Allows arbitrary policy logic via a custom evaluator function. The evaluator receives the full transaction context and returns an allow/deny decision.

{
  "type": "custom",
  "evaluator": "~/.lws/plugins/custom-policy.js"
}

The evaluator module MUST export a function with the following signature:

type PolicyEvaluator = (context: PolicyContext) => Promise<PolicyResult>;

interface PolicyContext {
  walletId: string;
  chainId: string;                   // CAIP-2
  transaction: SerializedTransaction;
  simulation?: SimulationResult;
  spendingHistory: SpendingRecord[];
  timestamp: string;                 // ISO 8601
}

interface PolicyResult {
  allowed: boolean;
  reason?: string;
  warnings?: string[];
}

Custom evaluators run in a sandboxed context with no network access and no access to key material. They MUST return within 5 seconds or the evaluation times out and the transaction is denied.

Evaluation Order

Rules are evaluated in a fixed order, regardless of their position in the policy's rules array. This ensures consistent, predictable behavior:

  1. Chain restriction — reject if the chain is not allowed.
  2. Denylist — reject if the target address is blocked.
  3. Allowlist — reject if the target address is not in the allowed set.
  4. Contract allowlist — reject if the contract is not in the allowed set.
  5. Require simulation — run simulation and reject if it fails.
  6. Max value — reject if cumulative spending exceeds the limit.
  7. Time restriction — reject if outside allowed hours.
  8. Custom — run custom evaluator(s) last.

Evaluation is short-circuiting: if any rule denies the transaction, subsequent rules are not evaluated.

Policy Actions

Each policy specifies an action that determines what happens when a rule is violated:

ActionBehaviorUse Case
denyTransaction is blocked. The signing process does not begin. An LwsError with code POLICY_DENIED is returned.Production policies, spending limits, security restrictions
warnTransaction proceeds but a warning is logged to the audit log and returned in the response. The agent or user can decide whether to continue.Monitoring, soft limits during development, gradual policy rollout

Policy Attachment

Policies are attached to wallets. Each wallet can have one active policy. Policies are stored as separate JSON files in ~/.lws/policies/ and referenced by ID in the wallet file.

CLI

# Create a policy
lws policy create --name "Agent Treasury Policy" --file policy.json

# Attach a policy to a wallet
lws policy attach --wallet agent-treasury --policy spending-limit-01

# View the active policy for a wallet
lws policy show --wallet agent-treasury

# Detach a policy
lws policy detach --wallet agent-treasury

Programmatic

// Attach a policy to a wallet
await lws.attachPolicy({
  walletId: "3198bc9c-6672-5ab3-d995-4942343ae5b6",
  policyId: "spending-limit-01"
});

// Create and attach inline
await lws.createPolicy({
  name: "Quick Limit",
  rules: [
    { type: "max_value", chainId: "eip155:8453", asset: "native",
      maxAmount: "500000000000000000", period: "24h" }
  ],
  action: "deny"
});

Spending State

The policy engine tracks cumulative spending in a local state file at ~/.lws/spending.json. This file is updated after every successful signing operation:

{
  "wallet:3198bc9c-6672-5ab3-d995-4942343ae5b6": {
    "eip155:8453:native": {
      "total": "450000000000000000",
      "entries": [
        {
          "amount": "200000000000000000",
          "timestamp": "2026-02-27T10:35:22Z",
          "txHash": "0xabc123..."
        },
        {
          "amount": "250000000000000000",
          "timestamp": "2026-02-27T14:12:08Z",
          "txHash": "0xdef456..."
        }
      ]
    }
  }
}

Entries older than the policy's period are pruned on each evaluation. The spending state file uses the same filesystem permissions as wallet files (600) and MUST NOT be world-readable.
